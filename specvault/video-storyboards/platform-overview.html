Platform Overview

The full SpecVault flow — from publishing a spec to closing the loop on a breaking change

~5:00
estimated duration
Target: API producers, platform & consumer teams
Scene 1 — Hook0:00 – 0:20

On Screen

Slack mock: "Why is the mobile app crashing? — Did someone remove ssn from the /users response?" Cut to a green CI checkmark next to it.

Voiceover

"Your build passed. Your tests are green. But a team downstream just found out in production that a field they depended on is gone. SpecVault catches that the moment the spec changes — and tells you exactly who it breaks, before it ships."

Scene 2 — What SpecVault is0:20 – 0:55

On Screen

Simple diagram: Publish spec → Version → Diff → Detect breaking changes → Notify affected consumers. Then the SpecVault dashboard home with a list of services.

Voiceover

"SpecVault is an API contract registry. Teams publish their OpenAPI specs, SpecVault versions them and diffs every new version against the last, and when something breaks, it alerts the consumers who actually rely on it. It's a registry and a diff engine — it doesn't run tests against your service, and there's nothing to install in your app."

Scene 3 — Register a service & publish the baseline0:55 – 1:35

On Screen

Create a service (name, owner, optional product). Open it, click Publish Version, paste a spec URL or upload openapi.json, set version 1.0.0. Show the parsed endpoint count on the Versions tab.

Voiceover

"Start by registering a service and publishing your first spec — by file, by URL, or straight from CI with an API token. SpecVault parses it, counts the endpoints, and stores it as version one-point-oh. This first version is the baseline: no alert fires on it — alerts only fire on diffs."

Scene 4 — Publish v2, the diff runs1:35 – 2:20

On Screen

Publish 2.0.0 where the User schema (behind a $ref) drops ssn and changes id from string to integer. Jump to the Changes tab — a new change event with 2 breaking.

Voiceover

"Publish a new version and the diff runs automatically. SpecVault resolves $ref and flattens allOf, so it compares the real shape of your schemas — through nested objects and arrays, across every response code. Here it catches two breaking changes a shallow text-diff would miss: a removed field and a type change, both defined in a shared component."

Scene 5 — Read the change2:20 – 2:55

On Screen

Expand the change event: red breaking rows ("response field 200.ssn removed", "type changed from string to integer") and green non-breaking rows. Open the "What SpecVault detects" disclosure briefly.

Voiceover

"Every change is classified direction-aware: tightening the request contract breaks producers, dropping or changing the response breaks consumers. The card shows exactly what changed and where. And the engine is honest about its limits — the coverage and known limitations are listed right here in the portal."

Scene 6 — Blast radius: who it breaks2:55 – 3:40

On Screen

Subscribers tab: payments@acme declared GET /users field ssn; analytics@acme declared a different field. Back on the change event: Blast radius — impacts payments@acme, analytics@acme spared, one subscriber "impact unknown".

Voiceover

"This is what sets SpecVault apart. Consumers declare the endpoints — and optionally the fields — they actually use. So a breaking change doesn't page everyone; it names the teams it truly affects. Payments reads ssn, so they're flagged. Analytics doesn't, so they're left alone. And anyone who hasn't declared their usage shows as 'impact unknown' — never silently assumed safe."

Scene 7 — Alerts3:40 – 4:10

On Screen

In-app notification bell with the breaking-change alert ("impacts 1 declared consumer, 1 unknown"), then the email version with the full diff table and a link back to the change.

Voiceover

"Subscribers are alerted in-app and by email — but only on breaking changes, so the signal stays high. Each alert carries the version jump, the full list of breaking changes, and the impact summary, with a link straight to the change event."

Scene 8 — Close the loop4:10 – 4:45

On Screen

On the change event: click Acknowledge (intentional) on one, Flag as Error on another → status "Fix Pending". Publish a corrective 2.0.1 → the flagged event flips to Resolved. Glance at the audit log.

Voiceover

"Breaking changes have a lifecycle. Acknowledge the ones that are intentional, flag the ones that aren't — and when you publish the fix, SpecVault auto-resolves the flag. Every action lands in the audit log, so there's always a record of what changed, who triaged it, and how it was closed."

Scene 9 — CI publish & pausing4:45 – 5:10

On Screen

A CI snippet doing POST /api/services/{id}/versions with a Bearer token after a build. Then a service being Deactivated (greyed, "Inactive") and reactivated.

Voiceover

"In practice you wire this into CI — publish the spec with an API token after every build, and breaking changes surface in review instead of in an incident. And when a service is winding down, deactivate it to pause alerts while keeping its full history; delete only when you truly want it gone for good."

Scene 10 — CTA5:10 – 5:30

On Screen

Timeline recap: Publish → Diff → Detect → Attribute → Notify → Resolve. URL: specvault.sentinelflux.in

Voiceover

"SpecVault turns the most common cause of integration outages — an undiscovered breaking change — into a review-time alert that names the teams affected. Publish, diff, detect, attribute, notify, resolve. Start at specvault.sentinelflux.in."
