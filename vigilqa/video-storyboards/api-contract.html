API Contract Testing

Consumer-driven contract tests — catch breaking changes before consumers do

~4:00
estimated duration
Target: platform engineers, API producers
Scene 1 — Hook0:00 – 0:20

On Screen

Slack message mock: "Hey, did you remove the user_id field from the orders response? Our mobile app is crashing in production."

Voiceover

"You've shipped a refactor. The API tests pass. But a consumer team finds out in production that a field they depended on is gone. API contract testing exists to prevent exactly this — catching breaking changes at the producer's CI/CD step, not in production."

Scene 2 — What contract testing does0:20 – 0:55

On Screen

Diagram: Consumer team defines what fields they need → Contract file generated → Producer's pipeline runs contract verification → Pass/Fail

Voiceover

"Consumer-driven contract testing flips the traditional model. Instead of the producer guessing what consumers need, consumers define exactly which fields and response shapes they depend on. VigilQA generates these contract definitions from your Knowledge Base and runs provider verification on every producer release."

Scene 3 — Contract KB scenario0:55 – 1:35

On Screen

KB editor: contract scenario with module: contract, provider and consumer names, the interaction — request + required response fields. Highlight the required_fields list.

Voiceover

"A contract scenario identifies the consumer, the provider, and the interaction — what request the consumer sends and which fields it requires in the response. The required_fields list is the heart of it: these are the fields that must be present and correctly typed in every provider release, or the contract breaks."

Scene 4 — Generated contract and verification1:35 – 2:20

On Screen

Scripts view: show the generated contract JSON (Pact-compatible format). Then show the provider verification test — it replays the contract against the live API and checks each required field.

Voiceover

"VigilQA generates a Pact-compatible contract file and a provider verification test. The verification test replays the consumer's expected interaction against the actual provider API and checks that every required field is present with the correct type. This runs in the producer's pipeline — the producer can't break a consumer without failing their own CI."

Scene 5 — Contract failure result2:20 – 3:05

On Screen

Failed contract run. Click to open: shows consumer name, the interaction that broke, which required field was missing or type-mismatched, and which provider release introduced the failure.

Voiceover

"When a contract breaks, the failure report names the consumer, the specific interaction, and exactly which field changed. There's no ambiguity about what broke or who is affected. The producer team can fix the issue or coordinate a migration with the consumer team before the change ships."

Scene 6 — CTA3:05 – 3:40

On Screen

Timeline: Producer ships → Contract verification runs → Breaking change caught → Consumer notified before prod. URL: vigilqa.sentinelflux.in

Voiceover

"API contract testing in VigilQA takes the most common cause of microservice production incidents — undiscovered breaking changes — and catches it at the earliest possible point. No Pact Broker required, no coordination overhead. Start at vigilqa.sentinelflux.in."
