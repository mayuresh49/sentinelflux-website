Visual Regression Testing

Pixel-level screenshot comparisons — catch UI regressions before users do

~4:00
estimated duration
Target: frontend engineers, product teams
Scene 1 — Hook0:00 – 0:20

On Screen

Side-by-side screenshots: before CSS refactor (clean layout) vs. after (button slightly shifted, nav link overlapping text). Highlight the difference with a red box. Text: "All functional tests passed."

Voiceover

"Functional tests tell you if the app works. They can't tell you if it looks right. A CSS refactor, a dependency update, a z-index change — all pass your test suite and still leave your users looking at a broken UI. Visual regression testing catches what code tests miss."

Scene 2 — How it works0:20 – 0:55

On Screen

Diagram: Baseline screenshot captured → New build runs → New screenshot taken → Pixel diff computed → Changes above threshold flagged

Voiceover

"Visual regression testing works by comparing screenshots. On the first run, VigilQA captures baseline screenshots of every page in scope. On every subsequent run, new screenshots are captured and compared pixel by pixel. Any difference beyond the configured threshold is flagged as a visual regression — with a diff image showing exactly what changed."

Scene 3 — KB scenario and config0:55 – 1:35

On Screen

KB editor: visual scenario with module: visual, page URL, viewport sizes listed (desktop 1440px, tablet 768px, mobile 375px), and threshold setting. Show multiple pages listed.

Voiceover

"A visual scenario simply lists the pages to capture and the viewport sizes to test at — desktop, tablet, and mobile. Configure your threshold: zero for exact pixel matching, or a small percentage to allow for font rendering variation. VigilQA handles the rest — navigating to each page, waiting for content to load, and capturing full-page screenshots."

Scene 4 — Visual results view1:35 – 2:20

On Screen

Visual results page: thumbnail grid showing all pages tested. Two thumbnails have red badges ("Diff: 2.3%"). Click one — opens full-screen side-by-side: baseline left, current right, diff highlighted in pink. Toggle between baseline and diff view.

Voiceover

"The visual results view shows every page tested as a thumbnail. Pages with differences are flagged with the diff percentage. Click any flagged page to see a full-size comparison — baseline on the left, current on the right, with the changed pixels highlighted. You can toggle to a diff-only view that isolates exactly what moved or changed."

Scene 5 — Approving baseline updates2:20 – 3:00

On Screen

Approvals view: visual diffs pending. Show a diff that's an intentional redesign. Click "Approve as new baseline". The thumbnail updates to the new screenshot. Show another diff that's clearly a regression — click "Reject".

Voiceover

"Not every visual diff is a regression — sometimes it's an intentional design change. The approvals workflow lets you decide: approve the new screenshot as the updated baseline, or reject it as a regression that needs fixing. Once approved, future runs compare against the new baseline. Your team stays in control of what 'correct' looks like."
