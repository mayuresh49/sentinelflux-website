LLM Configuration

SentinelFlux's agents use a large language model for generation and analysis. The default configuration requires no setup. Enterprise plans can connect a local Ollama model for air-gapped environments.

Default LLM (no configuration required)

By default, SentinelFlux uses a Mistral-based model hosted on SentinelFlux infrastructure. This model is optimised for code generation and structured reasoning tasks. No API key or LLM configuration is required from you — it is included in all plans.

What data is sent to the LLM?

During the generation pipeline, the following data is sent to the LLM:

  • Your Knowledge Base scenario descriptions (titles, steps, expected outcomes)
  • AppExplorer selector maps (DOM structure, element labels — not user data)
  • Previously generated test case documents (for ScriptGen context)

Test execution results, application user data, and database contents are not sent to the LLM. Post-run agent analysis (ResultAnalyzer, FlakyDetector) uses structured rule-based logic supplemented by the LLM only for the failure classification text — not the raw response data.

All LLM interactions use data scoped to your project's KB and structural metadata. Customer application data (user records, PII, financial data) is never included in LLM calls. See the Security page for full data handling details.

Local LLM via Ollama (Enterprise)

Enterprise plans support routing generation agent calls to a local Ollama instance instead of the SentinelFlux-hosted model. This is designed for organisations in air-gapped or strict data-residency environments where KB content must not leave the organisation's network perimeter.

With local LLM configured:

  • All generation pipeline LLM calls are routed to your Ollama instance
  • KB content and selector maps stay within your network boundary
  • Post-run agent classifications still use SentinelFlux-hosted logic unless the local option is also configured for analysis

Configuring Ollama

In Project Settings → LLM Configuration, select Local (Ollama) and provide:

  • Ollama base URL — the URL of your Ollama API endpoint (must be accessible from SentinelFlux's execution environment or your private network peering configuration)
  • Model name — the Ollama model identifier (e.g. qwen2.5-coder:32b)

Contact engage@sentinelflux.in to set up network peering for private Ollama access.
