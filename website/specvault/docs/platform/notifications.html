SpecVault Docs

Notifications

How SpecVault sends email alerts when breaking changes are detected, and how to configure your SMTP settings.

When alerts fire

SpecVault sends an email alert every time a new spec version is published and the diff against the previous version contains at least one breaking change. Additive-only changes (new endpoints, new optional fields) do not trigger alerts.

Each subscriber on the affected service receives one email per publish event. If a publish has no breaking changes, no email is sent.

Alert email format

The alert email contains:

  • Subject: [SpecVault] Breaking changes in <Service Name> v<version>
  • Service name and the version number that introduced the changes
  • Publish timestamp and the publisher's email
  • A numbered list of each breaking change, including the HTTP method, path, and a plain-English description (e.g. Required parameter "user_id" removed from GET /users)
  • A direct link to the diff detail page in the SpecVault dashboard
  • An unsubscribe link for that specific service

SMTP configuration

SpecVault uses a standard SMTP connection to send alerts. Set the following environment variables in your .env file before starting the application:

SMTP_HOST=smtp.example.com
SMTP_PORT=587
SMTP_USER=alerts@yourcompany.com
SMTP_PASSWORD=your-smtp-password
SMTP_FROM=alerts@yourcompany.com
SMTP_TLS=true

Provider examples

Common SMTP providers and their settings:

  • Gmail (App Password): host smtp.gmail.com, port 587, TLS true
  • SendGrid: host smtp.sendgrid.net, port 587, user apikey, password = your API key
  • AWS SES: host from SES console, port 587, SMTP credentials from IAM
  • Mailgun: host smtp.mailgun.org, port 587
  • Postmark: host smtp.postmarkapp.com, port 587

Gmail note: Standard Gmail passwords do not work over SMTP. You must create an App Password in your Google Account security settings and use that instead.

Testing your SMTP config

After setting environment variables and restarting, publish a spec version that introduces a breaking change (you can temporarily remove an endpoint from a test spec). If the subscriber address receives the alert, SMTP is working correctly.

Check application logs for SMTP errors if the email does not arrive. Common issues include wrong port, TLS mismatch, or firewall rules blocking outbound port 587.

No-email option

If SMTP is not configured, SpecVault still detects and stores breaking changes — it simply cannot send outbound alerts. The dashboard diff view will show breaking changes regardless of SMTP status. A warning badge appears in Settings when SMTP is unconfigured.

Tip: For local development, use MailHog or Mailpit as a local SMTP sink. They capture outbound emails without actually sending them.

Unsubscribe flow

Every alert email includes a one-click unsubscribe link. Clicking it removes the subscriber from that specific service only; they remain subscribed to any other services. Admins can also remove subscribers directly from the Subscribers tab in the dashboard.
