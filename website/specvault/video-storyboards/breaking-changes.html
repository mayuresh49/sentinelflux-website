Breaking-Change Detection

How the $ref-aware diff engine decides what's breaking — and what it deliberately ignores

~4:00
estimated duration
Target: API producers, platform engineers
Scene 1 — The shallow-diff trap0:00 – 0:25

On Screen

Two spec files side by side. The paths look identical — both point a response at $ref: '#/components/schemas/User'. The real change is inside components.

Voiceover

"Most real OpenAPI specs define their schemas once under components and reference them with $ref. A naïve diff that stops at the $ref sees no change here — and silently misses a breaking one. SpecVault doesn't stop there."

Scene 2 — Resolving the real shape0:25 – 1:05

On Screen

Animate the $ref expanding into the full User object; an allOf composition merging into one shape; a nested object and an array of objects unfolding.

Voiceover

"SpecVault resolves every $ref, flattens allOf composition into a single shape, and walks nested objects and array items to any depth — comparing across every response status code, not just 200. It compares the actual structure your consumers receive."

Scene 3 — Direction-aware severity1:05 – 1:55

On Screen

Split panel. Request (client → server): new required field, enum value removed, tighter maxLength → red. Response (server → client): field removed, type changed, new enum value, became nullable → red. Additive request changes → green.

Voiceover

"Severity depends on direction. Tightening the request — a new required field, a removed enum value, a stricter constraint — breaks the clients sending data. Dropping or changing the response — a removed field, a changed type, a value a consumer's switch won't handle — breaks the clients reading it. SpecVault scores each change from the right side of the contract."

Scene 4 — Deprecation & the diff card1:55 – 2:45

On Screen

A field gains deprecated: true → amber advisory row. Then the Changes tab card: breaking count in red, non-breaking in green, each row with method, path, and the dotted field path like 200.user.ssn.

Voiceover

"Marking something deprecated isn't breaking yet, but it's a warning shot — so SpecVault records it as an advisory change to give consumers a heads-up before the field disappears. Everything lands on the change card with a precise field path, so you know exactly what moved and where."

Scene 5 — Honest about limits2:45 – 3:30

On Screen

Open the "What SpecVault detects — and what it doesn't" disclosure on the Changes tab. Highlight: renames show as remove + add; oneOf/anyOf/discriminator not deeply compared; severity is a heuristic.

Voiceover

"And it tells you where it stops. A rename shows up as a removal plus an addition, not a paired 'rename'. oneOf and anyOf composition isn't deeply compared. Severity is a heuristic — so SpecVault always shows you the diff to review. No black box, no false confidence."

Voiceover

"A diff engine that reads your specs the way your consumers do — references resolved, shapes compared, severity scored by direction. That's how SpecVault catches the breaking changes that slip past everything else. specvault.sentinelflux.in."
