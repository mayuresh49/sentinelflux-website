Quality Thresholds

Configure the bands behind every Test Quality statistic — risk score, doc coverage, and test health — so the dashboard reflects your product's tolerance rather than a generic default.

Why configure thresholds

The defaults are tuned for a typical business application. A payments ledger, a medical device portal, or an authentication service is not typical. On those products a risk score of 30 is not "low" — anything above 15 warrants review, and a doc coverage figure of 70% is a gap, not a pass.

Thresholds are declared per project in the Knowledge Base YAML under a quality block. They change how values are classified and coloured across the dashboard, run summaries, and CI gates. They never change how the underlying scores are computed.

The quality block

Add the block at the top level of your Knowledge Base YAML. Every key is optional — omit a metric and it keeps its default bands.

quality:
  risk_score:          # lower is better
    low:    { max: 15 }
    medium: { max: 40 }
    high:   { max: 100 }

  doc_coverage:        # higher is better, percent
    green:  { min: 85 }
    amber:  { min: 60 }
    red:    { min: 0 }

  test_health:         # higher is better, percent
    green:  { min: 90 }
    amber:  { min: 75 }
    red:    { min: 0 }

Band direction

Two kinds of metric exist, and they are declared differently:

  • Lower-is-better metrics (risk_score) use max. A value falls into the first band whose max it does not exceed.
  • Higher-is-better metrics (doc_coverage, test_health) use min. A value falls into the first band whose min it meets.

Bands are evaluated best-first and are inclusive at the boundary. With the example above, a risk score of exactly 15 is low; 15.1 is medium. A doc coverage of exactly 85 is green; 84.9 is amber.

Validation rules

A malformed quality block fails the run at config load rather than silently falling back to defaults. The validator enforces:

  • Bands must be monotonic — each successive band's bound must be looser than the previous one.
  • The widest band must cover the end of the range (max: 100 or min: 0 for percent metrics), so every possible value lands somewhere.
  • A metric must use min or max consistently across all its bands, matching that metric's direction.
  • Unknown metric names and unknown band names are rejected, so a typo like doc_covrage surfaces immediately.

Scoping thresholds

Thresholds resolve most-specific-first: domain overrides project, project overrides the built-in defaults. This lets one strict domain sit inside an otherwise ordinary project.

quality:
  risk_score:
    low: { max: 30 }        # project-wide default

domains:
  - domain: payments
    quality:
      risk_score:
        low:    { max: 10 } # payments is held to a stricter bar
        medium: { max: 25 }
        high:   { max: 100 }
      doc_coverage:
        green: { min: 95 }

A partial override merges band-by-band. In the example the payments domain inherits the project's test_health bands untouched while replacing its risk and coverage bands.

Gating CI on bands

Because bands are named, CI gates reference the band rather than a duplicated number. Retune the threshold once and the gate follows.

quality:
  doc_coverage:
    green: { min: 85 }
    amber: { min: 60 }
    red:   { min: 0 }
  gate:
    warn_below: green       # amber marks the build unstable
    fail_below: amber       # red fails the build

See CI/CD for how unstable and failed states surface in each supported pipeline.

Environment-specific thresholds

Thresholds can differ per environment — useful when staging is expected to run hotter than production. Environment blocks override project-level values using the same merge rules.

environments:
  staging:
    quality:
      test_health:
        green: { min: 80 }  # relaxed while features land
  production:
    quality:
      test_health:
        green: { min: 95 }

See Environments for the full precedence order.

Worked example

A team shipping a critical product wants low risk to mean 15 or under, and coverage to be green only at 85% or above:

quality:
  risk_score:
    low:    { max: 15 }
    medium: { max: 40 }
    high:   { max: 100 }
  doc_coverage:
    green:  { min: 85 }
    amber:  { min: 60 }
    red:    { min: 0 }
  gate:
    fail_below: amber

With this in place, a scenario scoring 22 that previously displayed as low risk now displays as medium, and a suite at 78% doc coverage that previously showed green now shows amber — while a drop to 55% fails the build outright.
